This skill guides creation of distinctive, production-grade Flutter interfaces for iOS and Android phones. Build real, working, platform-aware code with strong aesthetic intent and polished interaction quality.

## Design Direction

Before coding, establish a clear direction:
- **Existing project first**: Reuse current theme, colors, typography, spacing, and component patterns in `lib/`.
- **Purpose**: What problem does this screen solve, and what mental state is the user in?
- **Platform target**: iOS, Android, or adaptive (both) with platform-specific behavior.
- **Tone**: Commit to a specific aesthetic (refined, playful, bold, utilitarian, editorial, dark/moody, vibrant, etc.).
- **The unforgettable element**: Define one signature visual or interaction moment.

**CRITICAL**: Respect existing project design patterns first, then platform conventions. Generic default Flutter styling is unacceptable.

## Flutter Mobile Design Guidelines

### Typography
→ *Consult [typography reference](reference/typography.md).*

Use intentional type hierarchy with readable body text and distinctive display treatment where appropriate.

### Color & Theme
→ *Consult [color reference](reference/color-and-contrast.md).*

Search and follow existing `ThemeData`, `ColorScheme`, `CupertinoThemeData`, and color constants before introducing new values.

### Layout & Space
→ *Consult [spatial reference](reference/spatial-design.md).*

Design for phones only. Prioritize thumb reach, safe areas, visual rhythm, and clear hierarchy over rigid symmetry.

### Motion
→ *Consult [motion reference](reference/motion-design.md).*

Use motion to communicate state, hierarchy, and continuity. Focus on a few high-impact moments.

### Interaction
→ *Consult [interaction reference](reference/interaction-design.md).*

Honor platform gestures and ergonomics. Build responsive controls with clear feedback and strong empty/error/loading states.

### Responsive & Device Adaptation
→ *Consult [responsive reference](reference/responsive-design.md).*

Support common phone widths and aspect ratios. Use adaptive layouts, not fixed dimensions.

### Accessibility & UX Writing
→ *Consult [ux writing reference](reference/ux-writing.md).*

Design for inclusivity: semantics, readable contrast, scalable text, reduced motion, and concise copy.

## Platform Expectations

### iOS
- Prefer `CupertinoApp`, `CupertinoNavigationBar`, `CupertinoTabScaffold`
- 44x44pt minimum touch targets
- Dynamic Type support and system-style transitions
- Respect swipe-from-left-edge back gesture
- Use semantic `CupertinoColors` with light/dark adaptation

### Android
- Prefer `MaterialApp`, `Scaffold`, `NavigationBar`
- 48x48dp minimum touch targets
- 8dp spacing rhythm and Material 3 component patterns
- Use `Theme.of(context)` and `ColorScheme.fromSeed()` where appropriate
- Respect system back and gesture expectations

---

## Phone-Only Constraints

- Always account for notches, cutouts, status bar, and home indicator via `SafeArea`/`MediaQuery`.
- Prioritize portrait. If landscape is supported, adapt layout intentionally.
- Test common phone widths (360dp, 375pt, 393pt, 412dp, 430pt).
- Keep primary actions in easy thumb zones (typically lower screen regions).

---

## Anti-Patterns

NEVER:
- Introduce new color systems when the app already has one
- Mix iOS and Android interaction patterns inappropriately
- Ship one-size-fits-all Flutter defaults with no design intent
- Use touch targets smaller than 44pt/48dp
- Ignore dark mode, safe areas, or text scaling
- Use animation as decoration without meaning
- Rely on generic AI aesthetics (safe card grids, default gradients, interchangeable style)

---

## The AI Slop Test (Mobile Edition)

If this screen could belong to any random app and immediately looks autogenerated, redesign it.

A strong result should feel:
1. Crafted for this specific product
2. Native to the target platform
3. Visually memorable in at least one intentional way

---

## Implementation Principles

- Match implementation complexity to design intent (maximal concepts need rich craft; minimal concepts need restraint and precision).
- Reuse project tokens/components before inventing new abstractions.
- Keep behavior production-safe: robust loading, error, and empty states.
- Ensure accessibility and platform correctness are first-class, not polish pass items.

Remember: Great mobile design balances creativity with platform familiarity. Build interfaces that feel both distinctive and native.
